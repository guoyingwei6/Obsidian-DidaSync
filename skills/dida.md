# Dida清单

使用 DidaSync MCP 工具管理 TickTick/Dida365 任务。先理解用户意图，再选择最小必要工具调用；不要用文件编辑代替 Dida 工具操作任务。

## 前置检查

1. 确认 MCP server 已配置为 `dida` 或可用的 DidaSync MCP server。
2. 如果工具不可用，提示用户在 DidaSync 设置中启用 MCP 服务，并在 YOLO 中配置：

```json
{
  "transport": "http",
  "url": "http://127.0.0.1:35829/mcp",
  "headers": {
    "Authorization": "Bearer <DIDASYNC_MCP_TOKEN>"
  }
}
```

3. 如果用户只是在询问任务内容，先读取任务；如果用户明确要求修改，再调用写工具。

## 编排原则（最重要）

- 先读后写：任何更新/完成/删除/排程前，必须先通过读取工具拿到真实 `id`/`didaId`。
- 先澄清再落库：候选任务不唯一、用户意图含糊、时间窗口不完整时，先问清楚再写入。
- 先给方案再写入：复杂安排先在回复里给“计划草案”，只有用户明确说“写入/更新/保存到滴答/同步”才执行写工具。
- 优先批量写入：涉及多任务时间规划，优先用批量排程工具一次性写入，避免多次单点更新导致中途失败或不一致。
- 最小变更：只更新用户要求的字段；不要顺手改标题/项目/优先级，除非用户说了。

## 常见流程

### 1) 任务查询与澄清（任何复杂操作的起点）

当用户问“今天有哪些任务”“某项目还有什么待办”“查一下 XX”：

1. 若用户只给了项目名而非项目 id：先列项目再筛任务。
2. 若用户给了关键词/标题片段：先搜索再让用户确认具体是哪一个任务（或哪个项目下的同名任务）。
3. 输出时按“可执行视图”总结：任务名、项目、状态、日期（有则写）、优先级（有则写）。
4. 这一步不做任何写入。

### 2) 新建任务（单任务，信息齐全就直接创建）

当用户说“帮我添加任务”“记一个待办”“创建 XX”：

1. 从用户话语中提取 `title`。
2. 若项目名不确定：先列项目给用户选；否则用收集箱。
3. 若日期/时间不确定：不要猜，问一个最关键问题（例如“哪天/几点开始，预计多久？”）。
4. 创建后只回显关键字段，避免贴完整 JSON。

### 3) 更新任务（单任务字段修改）

当用户说“把 XX 改成”“调整截止时间”“移动到某项目”：

1. 先定位任务：搜索或列表筛选，然后让用户确认选中哪一个（如果不唯一）。
2. 再更新：只带需要改的字段；涉及时间时，确保用户明确“开始/结束/是否全天”。
3. 更新后回显“改了什么”，并在需要时提示是否同步到滴答。

### 4) 每日时间规划（多任务时间块排程）

当用户说“帮我安排今天”“做一个今日计划”“把任务排到上午/下午/晚上”：

目标：把“未完成/逾期/今日相关”的任务，排进一个清晰的时间表（上午/下午/晚上或具体时段）。

流程：
1. 收集候选任务：先列出今日相关 + 逾期 + 未安排时间的任务（必要时分两次取数），并去重。
2. 约束收集（只问必须的）：可用时间段、硬截止、用户偏好（深度工作/会议/碎片）。
3. 生成草案：输出一个可执行日程表（时间块 + 任务），并标明“未安排/需要确认”的条目。
4. 用户确认：让用户明确“是否写入/是否同步/是否覆盖已有时间块”。
5. 批量写入：用批量排程把确认的任务写入开始/结束时间；未确认的不要写。
6. 复核：写入后再列一次今日任务，确认结果符合预期。

### 完成或删除任务

当用户说“完成 XX”“删掉 XX”：

1. 先定位任务并确认唯一性。
2. 完成可以直接执行；删除必须二次确认（明确说“删除/移除/不要了”）。
3. 批量完成/删除时，先给列表让用户确认范围，再执行。

### 同步

当用户说“同步滴答”“刷新任务”“从滴答拉最新”：

1. 调用 `dida_sync_now`。
2. 同步后如需展示结果，再调用 `dida_list_tasks`。

## 日期处理

- 用户说“今天/明天/下周”等相对日期时，按当前日期换算成具体日期。
- 有具体时间时使用带时区的 ISO 字符串，例如 `2026-05-25T18:00:00+08:00`。
- 只有日期、没有时间时设置 `isAllDay: true`。
- 不确定日期时不要猜测，先询问用户。

## 参数规则

- 不要发明工具 schema 里没有的参数。
- 写操作必须基于真实 `id`/`didaId`（从读工具结果中拿）。
- 时间规划优先批量排程；单点修正再用单任务更新。

## 输出要求

- 任务操作后简短确认，不输出完整 JSON，除非用户要求。
- 多任务列表优先用简洁清单：任务名、项目、日期、状态。
- 工具返回错误时，说明失败原因，并给出可执行修复建议，例如检查 DidaSync MCP 服务、token、OAuth 登录或网络。
